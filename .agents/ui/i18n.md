# Internationalization (i18n) Guide

This document explains how to use internationalization in the `@open20/ui` package and how to extend it in applications like `@open20/spellbook`.

## Architecture Overview

The i18n system uses a layered architecture:

```txt
@open20/spellbook (Application)
    extends
@open20/ui (Component Library)
    provides base translations
```

- `@open20/ui` provides base translations for shared UI components
- Apps like `@open20/spellbook` provide app-specific keys on top
- TypeScript generics provide typed access to extended translation shapes

## Overview

The UI package includes a built-in i18n system based on React Context.

- `I18nProvider` provides locale state + translation set
- `useTranslation()` returns a translation function `t`
- `useI18n()` returns full context (`t`, `locale`, `setLocale`, `translations`)

## Quick Start

### In `@open20/ui` components

```tsx
import { useTranslation } from '@open20/ui';

function MyComponent() {
  const t = useTranslation();
  return <button>{t('namespace.key')}</button>;
}
```

### In `@open20/spellbook`

```tsx
import { I18nProvider } from '@open20/ui';
import { enTranslations, zhCNTranslations } from './i18n';

<I18nProvider
  initialLocale="en"
  translationsSet={{
    en: enTranslations,
    'zh-CN': zhCNTranslations,
  }}
>
  <App />
</I18nProvider>;
```

## Setting Up `@open20/spellbook`

### Step 1: Define extended translations interface

Extend `BaseTranslations` to add app-specific namespaces and keys:

```ts
// packages/spellbook/src/i18n/index.tsx
import type { BaseTranslations } from '@open20/ui';

export interface SpellbookTranslations extends BaseTranslations {
  // Add app-specific top-level keys
  spells: string;
  searchSpells: string;

  // Add app-specific namespaces
  character: {
    create: string;
    edit: string;
  };
}
```

**Organization tip**: Group related keys under a namespace object rather than using flat dot-notation keys.

### Step 2: Create locale objects

Spread `defaultTranslations` and add app-specific keys:

```ts
// packages/spellbook/src/i18n/index.tsx
import { defaultTranslations } from '@open20/ui';

export const enTranslations: SpellbookTranslations = {
  ...defaultTranslations,
  spells: 'Spells',
  searchSpells: 'Search spells...',
  character: {
    create: 'Create Character',
    edit: 'Edit Character',
  },
};

export const zhCNTranslations: SpellbookTranslations = {
  ...zhCNTranslations,
  spells: '法术',
  searchSpells: '搜索法术...',
  character: {
    create: '创建角色',
    edit: '编辑角色',
  },
};
```

**Important**: Always use the spread operator (`...defaultTranslations`) to create a new object. Avoid `Object.assign(defaultTranslations, ...)` as it mutates the shared base object.

### Step 3: Use provider in app

```tsx
// packages/spellbook/src/App.tsx
import { I18nProvider } from '@open20/ui';
import { enTranslations, zhCNTranslations } from './i18n';

export function App() {
  return (
    <I18nProvider
      initialLocale="en"
      translationsSet={{
        en: enTranslations,
        'zh-CN': zhCNTranslations,
      }}
    >
      <YourApp />
    </I18nProvider>
  );
}
```

### Step 4: Use in components

```tsx
import { useTranslation } from '@open20/ui';

function SpellList() {
  const t = useTranslation();

  return (
    <div>
      <h1>{t('spells')}</h1>
      <button>{t('common.cancel')}</button>
      <p>{t('namespace.key')}</p>
    </div>
  );
}
```

## Translation Key Organization

### Namespace Structure

Organize keys by feature or component group using nested objects:

```ts
interface BaseTranslations {
  // Shared UI actions
  common: {
    cancel: string;
    save: string;
    // ...
  };

  // Feature-specific group
  featureName: {
    key1: string;
    key2: string;
  };

  // Another feature group
  anotherFeature: {
    // ...
  };
}
```

### Key Naming Conventions

1. **Use camelCase** for all key names
2. **Group related keys** under a namespace object
3. **Use descriptive names** that indicate the UI purpose
4. **Keep consistent** across locales

Example structure:

```ts
{
  common: {
    // Generic actions
    cancel: string;
    save: string;
    delete: string;
  },
  validation: {
    // Form validation messages
    required: string;
    minLength: string;
  },
  featureName: {
    // Feature-specific labels
    title: string;
    description: string;
  }
}
```

### Extending Base Translations

When adding app-specific translations:

1. Create an interface extending `BaseTranslations`
2. Add new namespaces or top-level keys
3. Provide all required keys in each locale object
4. Type your translations object with the extended interface

```ts
interface AppTranslations extends BaseTranslations {
  // New namespace
  newFeature: {
    title: string;
    description: string;
  };

  // New top-level key
  appName: string;
}
```

## Adding New Languages

`I18nProvider` currently supports `en` and `zh-CN` locale keys through `translationsSet`.

To add a new language:

1. Add the locale key to the `translationsSet` object
2. Create a translations object for that locale
3. Provide all keys from `BaseTranslations` plus any app-specific keys

```tsx
import { I18nProvider, defaultTranslations } from '@open20/ui';

const frTranslations = {
  ...defaultTranslations,
  common: {
    ...defaultTranslations.common,
    cancel: 'Annuler',
    save: 'Enregistrer',
  },
  // ... other namespaces
};

function App() {
  return (
    <I18nProvider
      initialLocale="en"
      translationsSet={{
        en: enTranslations,
        'zh-CN': zhCNTranslations,
        fr: frTranslations,
      }}
    >
      <YourComponents />
    </I18nProvider>
  );
}
```

## Using Translations with Parameters

The `t()` function supports parameter interpolation using `{paramName}` placeholders:

```ts
// In translations object
{
  clearFilters: 'Clear {count} filters',
  greeting: 'Hello, {name}!',
}

// In component
const t = useTranslation();
t('clearFilters', { count: 3 });  // "Clear 3 filters"
t('greeting', { name: 'Alice' }); // "Hello, Alice!"
```

Parameters are passed as the second argument to `t()`:

```tsx
function FilterButton({ count }: { count: number }) {
  const t = useTranslation();
  return <button>{t('clearFilters', { count })}</button>;
}
```

## Available Hooks

### `useTranslation`

Returns a translation function `t` with optional parameter interpolation:

```tsx
const t = useTranslation();

// Simple key lookup
const text = t('common.cancel');

// With parameters
const message = t('clearFilters', { count: 3 });
```

### `useI18n`

Returns full i18n context including locale state:

```tsx
const { t, locale, setLocale, translations } = useI18n();

// Change locale
setLocale('zh-CN');

// Access current locale
console.log(locale); // 'en' | 'zh-CN'
```

## Components Using i18n

The following shared components use translations. When adding new components, follow the same pattern:

- Form components: validation messages, labels
- Dialog components: button text, accessibility labels
- Data display components: empty states, status messages
- Interactive components: tooltips, action labels

## Best Practices

1. **Use translation keys** instead of hardcoded UI strings
2. **Organize keys by namespace** - group related keys under feature/component names
3. **Extend `BaseTranslations`** when adding app-specific translations for type safety
4. **Test layouts** in all supported locales (`en`, `zh-CN`)
5. **Use parameter interpolation** for dynamic content instead of string concatenation
6. **Keep key names descriptive** - they should indicate the UI purpose
7. **Spread, don't mutate** - always use `...defaultTranslations` when creating locale objects

## Troubleshooting

### Missing translation key

If a key is missing:

1. A warning is logged in the console
2. The key string is returned as fallback

To fix: Add the missing key to all locale translation objects.

### Hook used outside provider

`useTranslation()` and `useI18n()` must run under `I18nProvider`. Ensure your component tree is wrapped with the provider.

### Type errors when accessing custom keys

If TypeScript complains about custom keys:

1. Ensure you've extended `BaseTranslations` with your app-specific interface
2. Type your translations object with the extended interface
3. Pass the generic type to `I18nProvider<T>`

```tsx
<I18nProvider<SpellbookTranslations>
  initialLocale="en"
  translationsSet={{ en: enTranslations, 'zh-CN': zhCNTranslations }}
>
  <App />
</I18nProvider>
```
